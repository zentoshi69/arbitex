# Running ArbitEx 24/7 on a VPS

## 0. First-time setup (fix “no such file”, “not a git repository”, “no configuration file”)

**`/path/to/project` in tutorials is a placeholder.** You must run every `git` / `docker compose` command **inside the folder that contains `docker-compose.yml`**.

### A) Put the code on the server

**Option 1 — clone from GitHub** (if the repo is public or you have a token):

```bash
cd /root
git clone https://github.com/zentoshi69/arbitex.git
cd arbitex
```

If the repo is **private**, use SSH (`git@github.com:...`) or HTTPS with a [Personal Access Token](./GIT_PUSH_SETUP.md).

**Option 2 — copy from your Mac** (if you don’t use Git on the VPS):

```bash
# On your Mac (replace USER and VPS_IP):
rsync -avz --exclude node_modules --exclude .git "/Users/antoineantoniadistdu/Documents/CODE X/" root@YOUR_VPS_IP:/root/arbitex/
```

Then on the VPS:

```bash
cd /root/arbitex
```

### B) Confirm you’re in the right place

```bash
pwd
ls -la docker-compose.yml
```

You should see `docker-compose.yml`. If `ls` fails, you are **not** in the project root — do not run `docker compose` yet.

### C) Environment file

```bash
cp .env.example .env
# Edit .env (RPC URLs, keys, DATABASE_URL, REDIS_URL, etc.)
nano .env
```

Use production-oriented values; Docker Compose usually wires DB/Redis via service names — follow `.env.production.example` if present.

### D) Build and start (one command — saves time)

From the **repo root** (folder with `docker-compose.yml`):

```bash
sudo systemctl enable docker
sudo systemctl start docker
chmod +x scripts/vps-quick.sh
./scripts/vps-quick.sh --pull
```

- **`--pull`** — runs `git pull` when `.git` exists (skip if you only rsync code).
- **`--prod`** — uses `docker-compose.prod.yml` + `.env.production` (stricter env; needs keystore paths).
- **`--no-cache`** — full rebuild (slower; use after dependency upgrades).

Health is checked **inside** the `api` container (works even when port `3001` is not published on the host).

Manual equivalent without the script: `docker compose build api worker web && docker compose up -d`.

### E) Auto-deploy from GitHub (push a button — no SSH)

1. On the VPS, clone once: `cd /root && git clone … arbitex` (see section **A**).
2. In GitHub: **Settings → Secrets and variables → Actions** add:
   - `VPS_HOST`, `VPS_USER`, `VPS_SSH_KEY`, `VPS_DEPLOY_PATH` (e.g. `/root/arbitex`).
3. **Actions → Deploy to VPS → Run workflow**.

This runs `git pull` + `./scripts/vps-quick.sh` on the server. Optional checkbox **use_prod** matches `--prod`.

### F) If the project might already exist somewhere

```bash
find /root /home -name "docker-compose.yml" 2>/dev/null
```

`cd` to the directory that contains that file, then run **D** again.

## What “disconnect” means

- **Closing your laptop / losing SSH** does **not** stop Docker containers on the server if Docker is running with `restart: always` (see `docker-compose.yml`).
- **Snowtrace “Value”** (e.g. ~$0.19) is **only native AVAX** sent with the transaction. **Swap size and PnL** appear under **Token Transfers** and **Logs**, not in that field.

## 1. Docker stays up after reboot

On the VPS (Ubuntu/Debian):

```bash
sudo systemctl enable docker
sudo systemctl start docker
```

Deploy your stack from the project directory:

```bash
docker compose up -d --build
```

All services use `restart: always`, so Docker will bring them back after **reboot** or **crash**.

## 2. Optional: watchdog cron (belt-and-suspenders)

If you want a periodic “ensure stack is up” (e.g. after manual `docker stop`):

```bash
sudo crontab -e
```

Add (use **your** project directory — same folder as `docker-compose.yml`):

```cron
*/5 * * * * cd /root/arbitex && /usr/bin/docker compose up -d >/tmp/arbitex-compose.log 2>&1
```

## 3. SSH sessions vs long-running processes

- **SSH disconnect** only affects your shell session.
- For **one-off long commands** while logged in, use `tmux` or `screen` so you can reconnect.
- For **production**, rely on **Docker** + **systemd** (above), not a terminal tab.

## 4. Verify health

Inside the API container network, health is:

- `GET http://api:3001/health` (from other containers)
- On host: `curl -sS http://127.0.0.1:3001/health`

The compose file includes a **healthcheck** on the API service.

## 5. After code fixes (PnL / Snowtrace links)

Rebuild and redeploy so production matches the repo (from project root, e.g. `cd /root/arbitex`):

```bash
git pull
docker compose build --no-cache api worker web
docker compose up -d
```

**Note:** Old rows in the database may still show **pre-fix** `pnlUsd` until you backfill or only trust **new** trades after deploy.
